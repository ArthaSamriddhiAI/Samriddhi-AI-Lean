## 1.1 A2 output shape (verbatim from disk)

A2's persisted output is `A2Output` (`lib/agents/a2-classification.ts:92-99`). A3's primary input, keyed by holding via `holding_verdicts[].holding_ref`:

```ts
export type A2Output = {
  agent_id: "a2_classification";
  case_id: string;
  as_of_date: string;
  holding_verdicts: A2HoldingVerdict[];
  summary: A2Summary;
  reasoning_summary: string;
};

export type A2HoldingVerdict = {
  holding_ref: string;            // <-- A3 keys per-holding actions on this
  instrument_display_name: string;
  asset_class: string;
  sub_category: string;
  weight_pct: number;
  verdict: A2Verdict;             // <-- the per-holding verdict
  drivers: A2Driver[];
};

export type A2Verdict =
  | "maintain"
  | "monitor"
  | "discuss"
  | "review"
  | "unable_to_classify";         // <-- the fifth value; the sentinel (see 1.4)

export type A2Driver = {
  driver_type: A2DriverType;
  severity: A2DriverSeverity;     // "watch" | "flag" | "escalate"
  scope: A2DriverScope;           // "holding" | "portfolio_propagated"
  source_observation: string;     // links the driver to an observation category
  reason: string;                 // <-- the diagnostic prose (Layer 2 / LLM)
};

export type A2Summary = {
  maintain_count: number;
  monitor_count: number;
  discuss_count: number;
  review_count: number;
  unable_to_classify_count: number;
  one_line_characterization: string;   // <-- rollup prose (Layer 2 / LLM)
};
```

**Field carrying the per-holding verdict:** `A2HoldingVerdict.verdict` (`A2Verdict`). The product's four meeting-behaviour verdicts are `maintain | monitor | discuss | review`; `unable_to_classify` is a fifth, the sentinel.

**Field carrying the diagnostic prose:** three places, all written by A2's Layer 2 (LLM):
- `A2Driver.reason` — one sentence per driver (why this holding got this verdict);
- `A2Summary.one_line_characterization` — portfolio rollup;
- `A2Output.reasoning_summary` — agent-level reasoning trace.

**Implication for A3 (per-holding surface).** A3's per-holding actions consume `holding_verdicts`. The ping's "Monitor/Discuss/Review only, never Maintain" rule maps to: filter `verdict ∈ {monitor, discuss, review}`. `maintain` holdings need no action (no "so what"). `unable_to_classify` holdings should route to A3's sentinel, not to a recommendation. `holding_ref` is the join key. `drivers[].source_observation` already names the observation that drove the verdict, so A3 can ground a per-holding action in the same observation vocabulary A2 used.

## 1.2 M0.PortfolioRiskAnalytics observation shape (verbatim from disk)

It emits a structured **metrics** object, `PortfolioMetrics` (`lib/agents/portfolio-risk-analytics.ts:99-170`). The fixed 10-item observation vocabulary is produced one layer downstream, by `stitcher.ts:derivePreObservations` (`lib/agents/stitcher.ts:122-241`), as `PreObservation.vocab_candidate`.

Two further drift notes:
- `PreObservation.vocab_candidate` is typed as bare `string` (`stitcher.ts:51`), **not** an enforced literal union. The "fixed 10-item enum" is a product-thesis construct, not a type-level constraint anywhere in the live code.
- `sector_over_concentration`: M0 computes the underlying data (`sectorExposureMfLookThrough` with flag/escalate constants `SECTOR_FLAG_PCT=25`, `SECTOR_ESCALATE_PCT=35`) and A2 derives a `sector_concentration` driver from it internally (`a2-classification.ts:327-336`), but the stitcher's pre-observation list omits it. So sector concentration is observable to A3 via A2's drivers, not via the stitcher's `pre_observations`.

**The concentration breach data for the rebalance proposal's deterministic layer.** Located: `PortfolioMetrics.concentration.positionFlags` = `{ instrument, weightPct (current weight), severity: "flag" | "escalate" }`. Thresholds are exported numeric constants (`portfolio-risk-analytics.ts:35-38`): `POSITION_FLAG_PCT = 10`, `POSITION_ESCALATE_PCT = 15`, `SECTOR_FLAG_PCT = 25`, `SECTOR_ESCALATE_PCT = 35`. Model-band anchors are `MODEL_BANDS` (`:26`) and per-class `assetClass[*].{targetPct, band}`.

**What M0 already computes vs. what A3 must compute itself:**

| Quantity | Provided by M0? | Notes |
|---|---|---|
| Current weight of a flagged position | yes | `positionFlags[].weightPct` |
| Breach severity (flag/escalate) | yes | `positionFlags[].severity` |
| The threshold itself | yes (as constants) | `POSITION_FLAG_PCT` / `POSITION_ESCALATE_PCT` |
| Model band / target weight for a sleeve | yes | `assetClass[*].{targetPct, band}`, `MODEL_BANDS` |
| **Target weight for a trimmed position** | **NO** | A3 computes (e.g., trim to the threshold, or to a band-consistent target) |
| **Trim amount per glide-path step** | **NO** | A3 computes |
| **Number of glide-path steps** | **NO** | A3 computes |
| **Per-step trigger threshold** | **NO** | A3 computes |

So M0 supplies the *breach state* (current weight, threshold, severity); A3's deterministic layer must compute the *path from current to target* (target weight, per-step trims, step count, triggers). This is the deterministic core of the rebalance proposal that verify must assert.

## 1.3 Sibling-agent patterns + the two-layer question

| Sibling | Skill file | Impl file | Orchestrator wiring |
|---|---|---|---|
| A2 | `agents/a2_classification.md` | `lib/agents/a2-classification.ts` (in/out types `:92-138`; `classifyHoldings` `:276`; `runA2ReasonText` `:782`; `runA2Diagnostic` `:799`) | `pipeline.ts:297` `runA2Diagnostic({caseId, asOfDate, holdings, metrics, evidence})`, after S1 |
| risk-reward-stats | `agents/risk_reward_stats.md` | `lib/agents/risk-reward-stats.ts` (`RiskRewardOutput` `:104`; `RiskRewardInput` `:118`; `computeRiskReward` `:488`) | `pipeline.ts:129` `runRiskRewardDeterministic({...})`, ships data only (S1-bypass, ADR-0021) |
| portfolio-overlap | `agents/portfolio_overlap.md` (ref) | `lib/agents/portfolio-overlap.ts` (`PortfolioOverlapOutput` `:105`; `computePortfolioOverlap` `:345`; `templatedRollup` `:468`) | `pipeline.ts:147` `runPortfolioOverlapDeterministic({...})`, ships data only (ADR-0030) |

**The pointed question: does either sibling already mix a deterministic computation layer with an LLM-narration layer in one agent?**

**Yes — the two-layer (deterministic compute + LLM narration) pattern is the established house style.** A3 does not introduce it; it mirrors it.

- **A2** is explicitly two-layer (`a2-classification.ts:5-15` header; skill `## Two-Layer Operation`): Layer 1 `classifyHoldings` is pure/deterministic and owns the verdicts; Layer 2 `runA2ReasonText` is one Claude call that returns only `reason` strings, merged back in TypeScript. The merge (`runA2Diagnostic:807-836`) hard-requires a Layer-2 reason per driver and **Layer 2 cannot change a verdict**. This is the exact discipline A3 needs: the LLM voice wraps the computed numbers and cannot mutate them.
- **risk-reward-stats** is the closest *structural* precedent for A3's computed/narrated split, and makes the split legible at the type level:

```ts
// Layer 1 returns everything EXCEPT the narrated fields:
export function computeRiskReward(input: RiskRewardInput):
  Omit<RiskRewardOutput, "rollup" | "reasoning_summary" | "pms_aif_framework_notice"> { ... }

// The narrated layer carries its own provenance disclosure:
export type RiskRewardRollup = {
  text: string;
  generation_method: "templated" | "llm_fallback";   // <-- voice provenance disclosed on the output
  llm_fallback_trigger: string | null;
  is_synthetic_forward: boolean;
  synthetic_forward_disclosure: string | null;
};
```

- **portfolio-overlap** and **M0.PortfolioRiskAnalytics** are pure-deterministic (overlap's rollup is `generation_method: "templated"` only, no LLM; M0 is the feeder). So they are *not* mixed-layer.

**ADR question:** because A2 and risk-reward-stats already establish the mixed deterministic+LLM pattern, **A3 introduces no new architectural pattern, and no ADR is warranted for the pattern itself.** A3 should mirror risk-reward-stats' type-level split (computed fields grouped separately from a narrated block that discloses its `generation_method`) and A2's merge discipline (LLM returns prose keyed to computed rows; cannot mutate the numbers). An ADR may still be warranted for A3's *placement* (it is the first agent that reads another agent's output — A2's — rather than only metrics/evidence), to be decided at lock.

## 1.4 A2's sparse-input behaviour (sentinel discipline)

A2's defensive contract, verbatim (`a2-classification.ts:287-316`): when the metric foundation is genuinely absent, every holding gets a sentinel verdict plus a single typed driver naming the missing input:

```ts
if (!metrics) {
  const holding_verdicts = holdings.holdings.map((h) => ({
    ...,
    verdict: "unable_to_classify" as A2Verdict,
    drivers: [{
      driver_type: "evidence_unavailable" as A2DriverType,
      severity: "flag", scope: "holding",
      source_observation: "evidence_unavailable: M0.PortfolioRiskAnalytics metrics absent",
      facts: { missing: "metrics" },
    }],
  }));
  summary.unable_to_classify_count = holding_verdicts.length;
  ...
}
```

The skill states the discipline (`a2_classification.md:194-196`):
- **Edge case 1 (wrapper opacity / look-through unavailable):** normal data-coverage limit. "No sentinel is needed for this case."
- **Edge case 2 (evidence input genuinely unavailable):** "Verdict for affected holdings is `unable_to_classify`, with a driver of type `evidence_unavailable` citing which input is missing. This should be rare in production fixtures."

**The discipline A3 should mirror:** a sentinel is reserved for the genuine "cannot form an answer" case, surfaced as a *first-class typed value* (not an empty field or a null), carried in both the per-item shape and a `*_count` rollup in the summary, and it names *why* (which input/context is missing). It is **not** used for normal coverage limits. A3's honest non-answer — "recommendation not surfaced; no client-specific context" — is the advisor-action analogue of `unable_to_classify`, and should be representable on both the per-holding and per-observation surfaces (see schema proposal).

## 1.5 The three render surfaces + orchestrator confirmation

A3 produces three arrays in **one pass / one agent call** (not three runs): per-holding actions, per-observation actions, and the portfolio-level rebalance proposal.

**Orchestrator confirmation (the load-bearing wiring fact).** Both of A3's primary inputs are already in scope, sequentially, inside `runDiagnosticPipeline` (`lib/agents/pipeline.ts`):
- `metrics` (M0) computed at `:117`;
- `evidence` bundle in scope;
- `stitched` (carrying `pre_observations`) at `:254`;
- S1 briefing at `:277`;
- `a2Result = await runA2Diagnostic({...})` at `:297`, with `a2Result.output` (`A2Output`) available from `:304` onward.

**A3 slots in immediately after the A2 call (after `:311`, before `contentJson` assembly at `:320`)**, consuming `metrics`, `a2Result.output`, `evidence`, and optionally `stitched.pre_observations`. Its output then joins `contentJson` exactly as the siblings do today (`a2_classification`, `risk_reward_stats`, `portfolio_overlap` at `:325-328`) — add `a3_so_what: a3Result.output`. The S2 renderer reads only `briefing`; A3 ships as data only until T-5.09 design renders it (WA09). **Confirmed: a single A3 call placed after both A2 and M0 can be supplied both outputs without any orchestration restructuring.**

The per-observation surface ("one per M0 observation"): the cleanest deterministic source is `stitched.pre_observations` (the 7 live `vocab_candidate` items, each with a typed `payload`), supplemented by A2's `drivers[].source_observation` for the categories the stitcher omits (notably `sector_over_concentration`). This is a schema-lock decision (D4): does A3 key `observation_actions` off `pre_observations` (7 live), off the 10-item thesis enum (3 currently unproducible), or off the union surfaced through A2's drivers? Recommendation in the chat proposal.
