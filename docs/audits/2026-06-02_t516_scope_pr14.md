## Verdict

**T-5.16 is fully delivered in SCOPE (both parts, the coverage footnote, and the `PortfolioMetrics` fields all present; T-5.15 cleanly absent), BUT Part 2 carries a concrete correctness defect:** the PMS sector look-through reads the wrong field name (`s.sector` instead of `s.name`), so every disclosed-sector PMS aggregates its sector weight under a `null` sector. This is visible in the shipped fixtures (bhatt 14.52%, surana 5.44% of the sector roll-up labelled `null`). So the action is NOT only a PR-body clarification: it is a one-line deterministic code fix plus a free re-back-fill of the affected fixture metrics, AND a PR-body clarification to name T-5.16. Both are the primary's calls; this audit makes neither.

## Part-by-part (code evidence quoted)

### Part 1, per-stock weight roll-up: BUILT

`lib/agents/portfolio-risk-analytics.ts` adds `stockExposureLookThrough` (type at `:195`, computed at `:525-527`). The whole-book loop (`:488-524`) rolls each equity holding's stocks into a single per-instrument weight across direct + MF + PMS:
- direct: `addStock(h.instrument, w, "direct")` (`:492`),
- MF: `parseTopHoldings(fund["Top 5 Holdings (JSON)"])` then `addStock(r.name, w * r.weight_pct / 100, "mf")` (`:503,505`),
- PMS: `pc.top_holdings` then `addStock(r.name, ..., "pms")` (`:511,513`).

It is functional in the shipped cases: surana shows the Reliance-dominance number, `Reliance Industries 20.3% [direct]`; bhatt rolls direct + PMS (`Reliance 12.2% [direct]`, `Aditya Birla Capital 1.13% [pms]`). This is the data the deferred T-5.09 donut needs, present per WA9. Part 1 is built.

### Part 2, sector look-through completion: BUILT, but with a defect

`sectorExposureLookThrough` (type `:196`, computed `:528-530`) is a NEW whole-book sector roll-up that DOES include direct equity and PMS, not only MF:
- direct: `sectorOf(h.instrument)` then `addSector(sec.sector, w, "direct")` (`:495-496`), resolved via the T-5.14 sector_map,
- MF: `parseTopSectors(...)` then `addSector(s.sector, ..., "mf")` (`:504,506`),
- PMS: `addSector(s.sector, w * s.weight_pct / 100, "pms")` (`:514`).

Implementation note (not a gap): the literal spec was "remove the `if (!isMF) continue` exclusion at `:331`." The branch did NOT remove it; that exclusion still stands (`:439`) for the OLD MF-only field `sectorExposureMfLookThrough`, which is retained for the A2 contract (`a2-classification.ts`). Part 2's intent (direct + PMS in the sector aggregation) is instead delivered additively by the new `sectorExposureLookThrough` field. That is a defensible choice (preserve the A2 contract, add the whole-book view), and it does include PMS, so Part 2 is delivered.

**The defect (correctness, WA5/WA8).** The PMS sector path at `:514` reads `s.sector`, but the snapshot's PMS `top_sectors` entries are keyed `name`, not `sector`. Grounded: `snapshot.pms.funds[].portfolio_composition.top_sectors` entries read `{ "name": "Financial Services", "weight_pct": 34.82 }`. So `s.sector` is undefined and the sector weight aggregates under a `null` key. The PMS HOLDINGS path one line above (`:513`) correctly uses `r.name`, and the MF path is fine because `Top5SectorRow` genuinely carries `.sector` (`:268`); only the PMS SECTOR path uses the wrong name. Impact in the committed fixtures: bhatt's `sectorExposureLookThrough` shows `null: 14.52% [pms]` and surana shows `null: 5.44% [pms]`, where those weights should carry the PMS's disclosed sector names (Financial Services, Industrials, ...). Every case with a disclosed-sector PMS is affected. The fix is one line (`s.sector` to `s.name` at `:514`), deterministic and zero-spend, followed by a re-run of `backfill-risk-reward.ts` / re-computation of the affected fixture metrics (no agent, no API).

### Coverage footnote: BUILT

`lookThroughCoverage` (type `:197`, computed `:570-573`) carries, for stock and sector, `coveredWeightPct`, `uncoveredWeightPct`, and a footnote. Undisclosed PMS and opaque AIF route to uncovered with a footnote (`:531-536`): "Per-stock roll-up covers X% ...; Y% is not looked through (opaque AIF, undisclosed or unmatched PMS, ...)." Present and populated (bhatt stock covered 49.8% / uncovered 36%, footnote present; that 36% is the figure bhatt's prose cites).

### PortfolioMetrics fields: BUILT

The output type gained all three look-through fields: `stockExposureLookThrough` (`:195`), `sectorExposureLookThrough` (`:196`), `lookThroughCoverage` (`:197`), alongside the retained MF-only `sectorExposureMfLookThrough` / `mfCoverage`.

### T-5.15: cleanly ABSENT

No reference to T-5.15 in `lib/`, `scripts/`, `db/`, or `docs/decisions/`. It is deferred and not half-present; PR #14 does not touch it.

## PR-body accuracy (assess, not edited)

PR #14's body says "direct-equity sector map plus look-through wiring, so direct stocks contribute to sector attribution." That is accurate but UNDER-specifies T-5.16: it does not name T-5.16 by id, it omits Part 1 entirely (the per-stock true-weight roll-up, the Reliance-dominance data for the deferred donut), and it conflates the T-5.14 sector_map with the broader T-5.16 look-through (the sector_map unblocked Part 2's direct-equity sector contribution, but T-5.16 is the per-stock plus whole-book sector machinery, commit `30e522d`). Recommended (not made): name T-5.16 and describe both parts plus the coverage footnotes. But the PR-body wording is the lesser issue; the PMS-sector defect is the substantive one.

## Recommendation

T-5.16 is in PR #14 in full scope, so this is not a partial-landing question. The substantive finding is the Part 2 PMS-sector field-name defect (PMS sectors mislabelled `null`), a visible data-quality bug in a believability product that the deferred T-5.09 donut would render. Recommend fixing it before merge (one-line code change at `:514` plus a free deterministic re-back-fill of the affected fixtures), and clarifying the PR body to name T-5.16. The merge, the fix-then-merge-versus-merge-and-track call, and any PR-body edit remain the primary's (WA1/WA28). This audit holds; it changes no code, data, or the PR.
