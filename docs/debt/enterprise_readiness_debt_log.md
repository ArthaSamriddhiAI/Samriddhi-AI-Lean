# Enterprise readiness debt log

ET-series. Anticipatory ledger of architectural-seam considerations that activate at first paying client; the lean MVP roadmap does not build against ET entries. New per-series ledger, first instantiated v14 (2026-05-27); see `README.md` for the general per-series convention.

**Severity convention:** ET entries carry the tag `Deferred` (singular), not Low / Medium / High, because pre-activation severity is not meaningful.

**Activation trigger:** First paying client. The first task on activation is a seam audit against the live codebase at that time.

**Cross-series boundary note:** ET differs from all other series in being *anticipatory* (consider at activation, not in the current build phase) rather than *active* (close in the current build phase). ET entries are not in the lean MVP roadmap and do not block capability workstreams.

| ID | Description | Severity | Originating workstream | Target fix workstream |
|----|-------------|----------|------------------------|------------------------|
| ET1 | Tenant identity placement. Where `tenant_id` threads through schemas (cases, holdings, fixtures, model portfolios, evidence outputs, telemetry events, prompts, ADR-cited boundary docs). Samriddhi is effectively single-tenant today. Retrofitting at first paying client means a multi-table migration; adding a nullable column now and threading it through (defaulting to a sentinel value like "samriddhi-internal") would be cheap but is deliberately not done in the lean phase. First enterprise task: audit which schemas need `tenant_id` and what threading looks like. | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
| ET2 | Prompt and model version pinning beyond Git SHA. When a wealth firm asks "why did this recommendation change between Q1 and Q2," the answer needs to cite specific model and prompt versions. Currently the case-level append-only logging captures the prompt's file pointer and Git SHA; for the lean MVP that's sufficient. At enterprise scale, the prompt string captured with the case output (not just the file pointer) is the cleaner audit-trail shape, so a regenerated case can be reconciled against the historical record even if the file evolves. | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
| ET3 | Configuration boundary map. G2 SEBI rules come from a curated store (good). House view, model portfolio, IPS templates, materiality thresholds are in code or implicit. Each crosses a tenant-config boundary that will activate when firms customize. Audit: which constants in code become tenant-configurable, where do they live today, what's the surface area of the boundary. | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
| ET4 | Event and telemetry shape. Events should be queryable in stable shapes (case_id, tenant_id placeholder, agent_id, model_version, prompt_version, started_at, completed_at, cost_units) rather than freeform blobs. Audit: are the events Samriddhi emits today shaped queryably, or is this a refactor at first deployment? | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
| ET5 | Security posture statement v0. One-pager that answers vendor security questionnaire basics: data residency (where do prompts go, where does Anthropic store inputs), encryption at rest and in transit, retention policy, deletion-on-request, incident response, audit log access for the firm's own auditors. Draft before first sales conversation with a firm's CISO or compliance lead, not after. | Deferred | v14 enterprise readiness series | Draft before first enterprise sales conversation |
| ET6 | Compliance framework selection. SOC 2 Type II as polestar (de facto standard for B2B SaaS in regulated industries; Indian firms with US institutional LPs will ask). ISO 27001 cross-referenced for international cred. SEBI cybersecurity framework for regulated entities (the one that applies to clients and indirectly to vendors) read once and tagged. DPDP Act (India's data protection regime, narrower than GDPR but mandates consent, purpose limitation, breach notification) interacts directly with anonymization decisions in ET7. | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
| ET7 | Anonymization strategic posture. Edge redaction (swap PAN, name, email at the prompt boundary) vs full token-level redaction with a re-hydration map are architecturally different solutions. The choice determines which kinds of firms Samriddhi can land (some firms will not let identifiable client names go to Anthropic regardless of Anthropic's own data policy). Strategic posture decided before implementation; the implementation is then constrained by the posture rather than vice versa. | Deferred | v14 enterprise readiness series | Decide posture before first enterprise sales conversation |
| ET8 | Case identifier as integration surface. Samriddhi creates auditable diagnostic cases; the wealth advisor consults them. No approval/disapproval state, no case-lifecycle workflow, no senior-advisor sign-off (per the decision-support-not-decision-workflow principle). Samriddhi owns the case ID as a citable reference; the lifecycle of the case as a thing-being-acted-upon lives in the firm's CRM or ERP. Audit: what does the case ID look like, how does it cite cleanly across systems, does its format have any `tenant_id` leakage that would later complicate multi-tenancy. | Deferred | v14 enterprise readiness series | Activate at first paying client; seam audit |
